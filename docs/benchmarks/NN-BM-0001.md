# NN-BM-0001

# WallBench

Version 1

## Abstract

WallBench is a benchmark suite for evaluating coordination under semantic pressure. It measures whether a distributed cognitive system can maintain a stable objective while individual agents receive incomplete, delayed, or conflicting information. WallBench is the companion evaluation framework for the nyanya LLM architecture, designed to stress-test the OCML metabolic layer, the RES Guard, and the ACI inspection pipeline under adversarial input conditions.

## Status

Benchmark specification. Supersedes Draft 0.

References: [NN-TR-2026-001](https://github.com/nya-lab/NN-TR-2026-001).

## 1. Purpose

WallBench evaluates three core capabilities of a coordinated cognitive system:

1. **Coherence under fragmentation**: Can the system produce a coherent output when inputs are deliberately split, delayed, or reordered?
2. **Drift resistance**: Can the system avoid semantic collapse when forced to process sequential summaries of its own prior outputs?
3. **Conflict preservation**: Can the system maintain productive disagreement without reducing it to noise or collapsing to a false consensus?

## 2. Benchmark Families

### 2.1 Wall-Static

**Scenario**: Agents receive fixed partial evidence. Each agent sees a different, non-overlapping subset of the full problem context.

**Task Design**: A complex reasoning problem is decomposed into N non-overlapping fragments. Each fragment is assigned to a different OPU. The system must reconstruct the complete problem and produce a coherent solution.

**Evaluation**: The output is scored against a ground-truth solution that requires all fragments to be integrated. Partial solutions that omit fragment information are penalized.

**Difficulty Tiers**:

| Tier | Fragment Count | Overlap | Time Pressure |
|---|---|---|---|
| Static-1 | 3 | 0% | None |
| Static-2 | 5 | 0% | Moderate |
| Static-3 | 8 | 10% (decoy) | High |

### 2.2 Wall-Drift

**Scenario**: Agents receive sequential summaries of prior agent outputs. Each round's input is the output of the previous round, simulating a multi-hop relay chain.

**Task Design**: A seed document is given to OPU-1. OPU-1 produces a summary. OPU-2 receives only the summary (not the original) and produces its own summary. This continues for K rounds.

**Evaluation**: The K-th round output is compared to the original seed document. The primary metric is semantic drift — the distance between the original and the final summary.

**Difficulty Tiers**:

| Tier | Relay Length (K) | Seed Complexity | Compression Ratio |
|---|---|---|---|
| Drift-1 | 3 | Low | 0.5 |
| Drift-2 | 5 | Medium | 0.3 |
| Drift-3 | 8 | High | 0.2 |

### 2.3 Wall-Conflict

**Scenario**: Agents receive evidence that supports incompatible interpretations. The system must preserve disagreement without turning it into noise.

**Task Design**: A single problem is presented with two conflicting evidence sets (A and B). Half the OPUs receive evidence set A, half receive set B. The system must produce an output that acknowledges the conflict, preserves both perspectives, and quantifies the uncertainty.

**Evaluation**: The output is scored on: (1) whether both interpretations are represented; (2) whether the conflict is explicitly acknowledged; (3) whether the uncertainty is quantified; (4) whether the system avoids collapsing to a single false resolution.

**Difficulty Tiers**:

| Tier | Conflict Severity | Evidence Symmetry | Domain |
|---|---|---|---|
| Conflict-1 | Low | Balanced | Factual |
| Conflict-2 | Medium | Skewed (70/30) | Analytical |
| Conflict-3 | High | Balanced | Ethical |

### 2.4 Wall-Noise

**Scenario**: Agents receive high-quality signals mixed with deliberately injected noise. The system must filter noise while preserving valid signals.

**Task Design**: A set of valid OSU signals is combined with noise signals (high-entropy, low-eta, or deliberately misleading). The noise ratio is configurable.

**Evaluation**: Precision and recall of valid signal retention. False positives (noise treated as signal) and false negatives (signal treated as noise) are both penalized.

**Difficulty Tiers**:

| Tier | Noise Ratio | Noise Type | Valid Signal Count |
|---|---|---|---|
| Noise-1 | 30% | Random | 10 |
| Noise-2 | 50% | Adversarial | 10 |
| Noise-3 | 70% | Mixed | 5 |

### 2.5 Wall-Latency

**Scenario**: Agents operate under heterogeneous latency profiles. Some produce fast, shallow answers; others produce slow, deep answers. The system must balance speed and depth.

**Task Design**: A cluster is configured with a mix of fast-shallow OPUs (tau ~ 100ms) and slow-deep OPUs (tau ~ 2000ms). The system must produce a merged output within a global deadline.

**Evaluation**: Output quality at different deadline thresholds. The primary metric is the quality-latency curve — how much quality is gained per additional unit of latency budget.

**Difficulty Tiers**:

| Tier | Fast OPU Ratio | Slow OPU Ratio | Global Deadline |
|---|---|---|---|
| Latency-1 | 70% | 30% | 5000 ms |
| Latency-2 | 50% | 50% | 3000 ms |
| Latency-3 | 30% | 70% | 1500 ms |

## 3. Metrics

### 3.1 Primary Metrics

`K`

Coordination efficiency. The ratio of useful output information to total communication volume across the cluster. Higher K indicates more efficient coordination.

```
K = I_useful / V_communication
```

`D`

Semantic drift. The semantic distance between the original input and the final output after passing through the coordination pipeline. Lower D indicates better semantic preservation.

```
D = semantic_distance(input_original, output_final)
```

`R`

Recovery after contradiction. The time (in rounds) required for the system to return to a stable consensus after a contradiction is introduced. Lower R indicates faster recovery.

`P`

Provenance retention. The fraction of input signal information that is traceable in the final output's provenance chain. P = 1.0 indicates full traceability.

```
P = |provenance_ids_retained| / |provenance_ids_available|
```

### 3.2 Secondary Metrics

`CHC_rate`

Cascade High-Confidence Cognitive Bias rate. The fraction of test cases where the system produces a false high-confidence consensus. Lower is better.

`ESL_count`

Echo Semantic Loop count. The number of detected ESL events per test run. Lower is better.

`RES_peak`

Peak Recursive Execution Saturation. The maximum RES value observed during a test run. Higher values indicate more severe scheduling degradation.

`BES_peak`

Peak Base Execution Saturation. The maximum BES value observed during a test run. Higher values indicate more severe execution stagnation.

`DSR_backlog`

Deferred Semantic Residue backlog. The number of DSR fragments awaiting reclamation at test completion. Lower is better.

## 4. Scoring Methodology

### 4.1 Composite Score

The overall WallBench score is a weighted composite:

```
WallScore = w1 * (1 - D_norm) + w2 * K_norm + w3 * (1 - R_norm) + w4 * P_norm + w5 * (1 - CHC_rate)
```

where each metric is normalized to [0, 1] against benchmark baselines, and weights default to:

| Weight | Value | Rationale |
|---|---|---|
| w1 | 0.25 | Drift resistance is the primary capability |
| w2 | 0.20 | Coordination efficiency |
| w3 | 0.15 | Contradiction recovery |
| w4 | 0.10 | Provenance traceability |
| w5 | 0.30 | Bias resistance is weighted highest |

### 4.2 Tiered Scoring

Each benchmark family is scored independently. The overall WallScore requires passing all families at the specified tier:

| Tier | WallScore Threshold | Description |
|---|---|---|
| Bronze | >= 0.60 | Basic coordination capability |
| Silver | >= 0.75 | Reliable coordination under moderate pressure |
| Gold | >= 0.85 | Robust coordination under high pressure |
| Platinum | >= 0.92 | Near-optimal coordination under extreme pressure |

## 5. Test Harness

### 5.1 Configuration

```
WallBenchConfig {
  families:        Family[]     // benchmark families to run
  tier:            Tier         // difficulty tier (1-3)
  repeat:          uint         // repetitions per test case
  timeout:         ms           // global timeout per test case
  opu_cluster_size: uint        // number of OPUs in test cluster
  seed:            uint         // random seed for reproducibility
}
```

### 5.2 Execution Flow

```
run_wallbench(config) -> WallBenchResult:
  1. initialize OPU cluster with config.cluster_size
  2. for each family in config.families:
     a. generate test cases for config.tier
     b. for each test case:
        - reset cluster state
        - execute test case
        - collect metrics
     c. aggregate family-level metrics
  3. compute composite WallScore
  4. return WallBenchResult
```

### 5.3 Result Schema

```
WallBenchResult {
  wall_score:         scalar      // composite score [0, 1]
  tier_achieved:      Tier        // highest passing tier
  family_scores: {
    wall_static:      scalar
    wall_drift:       scalar
    wall_conflict:    scalar
    wall_noise:       scalar
    wall_latency:     scalar
  }
  primary_metrics: {
    K:                scalar
    D:                scalar
    R:                scalar
    P:                scalar
  }
  secondary_metrics: {
    CHC_rate:         scalar
    ESL_count:        uint
    RES_peak:         scalar
    BES_peak:         scalar
    DSR_backlog:      uint
  }
  metadata: {
    config:           WallBenchConfig
    duration:         ms
    total_signals:    uint
    total_rounds:     uint
  }
}
```

## 6. Relationship to L8B

WallBench is the evaluation framework; L8B (NN-TR-2026-001 §9) is the input dataset. The relationship is:

- **L8B** provides the eight categories of extreme interference inputs (conflicting facts,残缺 text, modal contradictions, implicit semantics, high-emotion instructions, pollution context, meaningless high-confidence noise, multi-round iterative rewrites).
- **WallBench** defines the evaluation scenarios, metrics, scoring methodology, and reporting format that consume L8B inputs.

A complete evaluation runs WallBench across all five benchmark families, each using L8B inputs as the underlying test material.

## 7. Reporting Rule

WallBench results are fictional unless backed by a runnable implementation and clearly labeled as measured output.

Every WallBench report must include:

1. The exact configuration used (seed, cluster size, tier, repeat count).
2. Raw metric values for each test case (not just aggregates).
3. A provenance chain for the composite WallScore, tracing back to individual test case results.
4. Disclosure of any test cases that timed out or produced invalid outputs.
